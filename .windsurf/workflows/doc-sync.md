# Doc-Sync Workflow

Этот workflow обеспечивает консистентность между документацией и кодом.

## Когда использовать

- При изменении кода, затрагивающего функциональность из Story
- При изменении дизайна или архитектуры
- При добавлении новой фичи
- Перед merge в main ветку

## Шаги

### 1. Проверить текущее состояние
```bash
npm run docs:check
```

### 2. Определить затронутые документы

Открой `docs/TRACEABILITY.md` и найди:
- Какие Stories связаны с изменёнными файлами
- Какие Epics содержат эти Stories

### 3. Обновить Story (если нужно)

Если изменение влияет на Acceptance Criteria:

1. Открой файл Story в `docs/stories/S-XX-название.md`
2. Обнови секцию **Acceptance Criteria**
3. Обнови секцию **Файлы** если добавились новые файлы
4. Измени статус если нужно

### 4. Обновить Epic (если нужно)

Если изменились критерии готовности Epic:

1. Открой файл Epic в `docs/prd/epic-XX-название.md`
2. Обнови чекбоксы в **Критерии готовности**
3. Обнови таблицу Stories если добавились новые

### 5. Обновить TRACEABILITY.md

**ОБЯЗАТЕЛЬНО** после любых изменений:

1. Открой `docs/TRACEABILITY.md`
2. Обнови статусы Stories
3. Обнови связанные файлы если изменились
4. Обнови дату в таблице "Документ → Зависимости"
5. Добавь запись в Changelog

### 6. Обновить ROADMAP.md (при значимых изменениях)

Если изменился статус Epic или Story:

1. Открой `docs/ROADMAP.md`
2. Обнови таблицу статусов
3. Обнови дату "Last Updated"

### 7. Финальная проверка
```bash
npm run docs:check
```

Убедись что нет ошибок перед коммитом.

## Быстрые команды

| Действие | Команда |
|----------|---------|
| Проверить консистентность | `npm run docs:check` |
| Открыть трассировку | `code docs/TRACEABILITY.md` |
| Найти Story по файлу | `grep -r "имя_файла" docs/stories/` |

## Правила

1. **Никогда** не коммить код без проверки `docs:check`
2. **Всегда** обновлять TRACEABILITY.md при изменении связей
3. **Один** источник истины — TRACEABILITY.md
4. При конфликте между документами — TRACEABILITY.md главнее
